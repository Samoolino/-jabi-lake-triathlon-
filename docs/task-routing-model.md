# Task Routing Model

## 1. Purpose

The **Task Routing Model** defines how the AI-Agent Workforce Operating System behaves in execution. It translates the marketplace-core architecture into a practical routing engine for the Jabi Lake AI-ESG Triathlon.

This document explains how tasks are created, classified, prioritized, assigned, escalated, reassigned, and closed under Tarkwa’s supervision.

The goal is to move the system from concept into operational logic.

---

## 2. Routing Objective

The routing model exists to ensure that:

- the right task reaches the right agent at the right time
- urgent and high-impact work is surfaced quickly
- overloaded agents do not become systemic bottlenecks
- dependencies are visible before failure occurs
- ESG-sensitive and safety-sensitive work receives proper controls
- humans are involved when needed
- the system remains measurable and auditable

---

## 3. Routing Engine Overview

The routing engine is the coordination mechanism between:

- the **Task Registry**
- the **Agent Registry**
- the **Event State Engine**
- the **Scoring Engine**
- and **Tarkwa** as guardian orchestrator

### Routing sequence

```text
Signal/Input -> Task Creation -> Classification -> Priority Scoring -> Fit Matching -> Assignment -> Execution Monitoring -> Completion or Escalation
```

---

## 4. Task Intake Sources

Tasks may enter the system from six primary sources.

### 4.1 Scheduled tasks
Predefined operational tasks that are triggered by timeline or checklist state.

Examples:
- pre-race venue check
- volunteer coverage check
- transition area setup verification

### 4.2 Sensor-triggered tasks
Tasks generated by live environmental or venue intelligence signals.

Examples:
- air quality threshold exceeded
- route moisture increase detected
- crowd density spike near transition area

### 4.3 Human-requested tasks
Tasks submitted by operators, event leads, volunteers, partners, or officials.

Examples:
- request for route update
- missing signage report
- sponsor activation support request

### 4.4 Dependency-triggered tasks
Tasks created automatically because another task requires a downstream action.

Examples:
- air quality alert creates executive summary task
- blocked transport route creates alternative access analysis task

### 4.5 Incident-triggered tasks
Tasks created due to operational anomalies or critical deviations.

Examples:
- athlete support incident
- emergency route obstruction
- communications inconsistency during live updates

### 4.6 Governance-triggered tasks
Tasks created to satisfy compliance, ESG, legal, or audit requirements.

Examples:
- waste visibility verification
- compliance review for restricted venue action
- audit capture of critical decision sequence

## 5. Task Classification Logic

Every new task must be classified before routing.

### 5.1 Required classification fields
Each task should be assigned:
- task type
- domain
- urgency level
- criticality level
- ESG sensitivity
- human review requirement
- dependency count
- expected output type
- timeout condition

### 5.2 Task type classes
- planning
- sensing
- reporting
- validation
- coordination
- escalation
- recovery
- governance

### 5.3 Domain classes
- governance
- race operations
- environment
- accessibility
- logistics
- reporting
- marketplace
- data systems

---

## 6. Priority Scoring Model

Every task must receive a priority score before assignment.

## 6.1 Priority factors
Use these seven factors:
- urgency
- operational criticality
- safety relevance
- ESG sensitivity
- public visibility
- dependency pressure
- timing window risk

## 6.2 Illustrative formula

```text
Priority Score =
(urgency x 0.20) +
(operational criticality x 0.20) +
(safety relevance x 0.20) +
(ESG sensitivity x 0.10) +
(public visibility x 0.10) +
(dependency pressure x 0.10) +
(timing window risk x 0.10)
```

### 6.3 Priority bands
- **P1 Critical**: immediate action required
- **P2 High**: action required in near-real time
- **P3 Standard**: normal operational handling
- **P4 Deferred**: can wait without meaningful risk

---

## 7. Agent Fit Matching

After classification and priority scoring, the routing engine determines the best-fit agent.

## 7.1 Matching criteria
Each task should be matched against:
- capability fit
- domain fit
- current state availability
- current workload
- reliability score
- recent performance trend
- escalation history
- human-review requirement

## 7.2 Illustrative fit formula

```text
Agent Fit Score =
(capability fit x 0.30) +
(domain fit x 0.15) +
(state readiness x 0.15) +
(reliability x 0.15) +
(workload balance x 0.10) +
(recent performance trend x 0.10) +
(governance suitability x 0.05)
```

### 7.3 Assignment rule
The routing engine should assign the task to the highest-fit eligible agent unless:
- a human-review rule blocks autonomous routing
- a dependency is unresolved
- the target agent is overloaded
- Tarkwa overrides the default route

---

## 8. Routing Modes

The system should support multiple routing modes.

### 8.1 Direct routing
Single task -> single agent

Used when:
- task is clear
- domain is narrow
- no major dependencies exist

### 8.2 Cluster routing
Single task -> cluster lead -> functional agent

Used when:
- issue affects multiple functions within one domain
- cluster-level prioritization is needed

### 8.3 Collaborative routing
Single task -> multiple agents

Used when:
- the task needs joint execution
- one agent senses and another reports
- one agent validates another’s work

Example:
- Moisture Agent detects wetness
- Surface Condition Agent validates route risk
- Live Report Agent publishes operational summary

### 8.4 Escalation routing
Task bypasses normal route and moves up hierarchy

Used when:
- safety thresholds are exceeded
- legal or governance risk is material
- multiple clusters are affected
- event continuity is threatened

---

## 9. Tarkwa in the Routing Engine

Tarkwa is the supervisory protocol that governs routing exceptions, cross-domain task logic, and high-impact decision flow.

## 9.1 When Tarkwa intervenes directly
Tarkwa should intervene when:
- task priority is P1 Critical
- two or more clusters are implicated
- dependency chains become unstable
- agent confidence drops below threshold on high-impact tasks
- a human escalation trigger is reached
- assignment conflict cannot be resolved automatically

## 9.2 Tarkwa routing actions
Tarkwa may:
- approve a route
- reroute a task
- split a task into subtasks
- require validation before completion
- escalate to human oversight
- force cross-cluster coordination
- suspend low-priority tasks during critical load

---

## 10. Execution Monitoring Logic

Routing is not complete at assignment. The engine must monitor execution.

## 10.1 Assignment checkpoints
After assignment, the system should track:
- acceptance timestamp
- task start timestamp
- state changes
- dependency resolution status
- output submission
- timeout breaches
- escalation events

## 10.2 Watch conditions
A task should be watched more closely if it is:
- high priority
- safety sensitive
- ESG sensitive
- public-facing
- multi-agent
- near deadline

---

## 11. Escalation Model

Escalation exists to protect the system from silent failure.

### 11.1 Escalation triggers
Escalate when:
- no acceptance occurs within timeout
- agent enters blocked state
- output confidence falls below threshold
- dependency chain stalls
- safety relevance increases during execution
- task becomes cross-domain
- public or governance risk becomes material

### 11.2 Escalation levels
- **E1 Functional escalation**: to cluster lead
- **E2 Cluster escalation**: to Tarkwa
- **E3 Strategic escalation**: to Tarkwa + multiple cluster leads
- **E4 Human escalation**: to event command / human operators

---

## 12. Reassignment Logic

Reassignment should be explicit and auditable.

### 12.1 Reassignment conditions
- target agent times out
- target agent becomes overloaded
- target agent loses required confidence level
- task context changes significantly
- higher-fit agent becomes available
- Tarkwa overrides assignment

### 12.2 Reassignment process

```text
Detect failure or mismatch -> Mark current assignment as blocked or failed -> Recompute fit scores -> Assign new agent -> Record reason
```

---

## 13. Multi-Agent Coordination Logic

Some tasks should be decomposed into subtasks.

### 13.1 Decomposition rules
A task should be split when:
- more than one capability is required
- output requires validation before publication
- one task contains both sensing and action
- the issue affects multiple operational zones

### 13.2 Parent-child task model
Use:
- parent task
- subtask A
- subtask B
- validation task
- reporting task

Example:

```text
Parent Task: Route moisture risk at bike segment east
  -> Subtask A: Moisture Agent verifies wetness
  -> Subtask B: Surface Condition Agent scores slip risk
  -> Subtask C: Event Operations Agent checks reroute readiness
  -> Subtask D: Live Report Agent drafts update
  -> Validation: Tarkwa or cluster lead confirms action set
```

---

## 14. Human-in-the-Loop Controls

The system must remain accountable.

### Human approval required for:
- event suspension or restart
- medical or life-safety actions
- legal or regulatory actions
- sensitive public incident statements
- sponsor-critical public messaging
- major access shutdown decisions

In these cases, the routing engine may prepare recommendations, but must not finalize action without human confirmation.

---

## 15. Routing State Machine

```text
New -> Classified -> Scored -> Matched -> Assigned -> Accepted -> Active -> Completed
```

Alternative paths:

```text
Assigned -> Timeout -> Escalated -> Reassigned
Assigned -> Blocked -> Escalated -> Split into subtasks
Active -> Confidence drop -> Validation required
Completed -> Review -> Scored -> Archived
```
